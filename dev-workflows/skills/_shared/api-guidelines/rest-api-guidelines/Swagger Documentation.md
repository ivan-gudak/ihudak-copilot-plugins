# Introduction & Goals
API documentation is provided in the form of OpenAPI documents exposed in a Swagger UI hosted on the API Gateway. The specific platform services serve the documents, and the API Gateway adapts the documents to match the external representation (paths, URLs, etc.).

Many services contribute to the overall "Dynatrace Platform API" which are all maintained by different development teams. **It is crucial to conform to common structural guidelines regarding the service documentation to provide a consistent overall API experience for customers**. This includes naming schemas, versioning formats, general structure and common shared resources where possible.

Although endpoints are essentially APIs on K8s services we call them "services" on the outside. So e.g., the public App Registry API is published as "App Registry Service". This system is described in more details in the [General Structure](../rest-api-guidelines/General%20Structure.md) section.

# "Select a Definition" - The Service List
This is the starting point of a customer and the first impression of the Dynatrace Platform API:
![The Platform Service List](../rest-api-guidelines/img/service%20list.png)

First and foremost: **it is a list of services, not a list of APIs!** See [General Structure](../rest-api-guidelines/General%20Structure.md) for details.

List entries **must** follow these rules:

- Specify a service name, not an operation (the name must be approved by PM)
- Avoid redundant postfixes like "API" or "Service", stick to the real name
- Do not show the service version if there is only one version, only show the version in the list of there are multiple versions of the service available

# The Service Name
Service names **must** follow these rules:

- Use the same name in all relevant locations
    - Service List
    - Title
    - URLs (OpenAPI document path and server path) - a shortened version of the name **may** be used for the sake of simplicity
- Translate names that consist of multiple words to kebab-case
    - E.g., "App Registry" translates to "app-registry" in the URLs

![Service Name Locations](../rest-api-guidelines/img/consistent%20naming.png)

The service version **must** be the last part in the _servers_ URL. It **must not** be part of the _paths_ section of the document.

# Consistent Document Content
## Overview
Each service **must** at least provide an _Overview_ in the general description which briefly describes the service, its purpose and general functionality. It **should** be kept short and **may** contain links to official external Dynatrace documentation pages (e.g., if referencing to general DQL Syntax description). The Overview **should** make use of markdown for better formatting. Additional sections **may** be added but **should** be kept short.

Example for the _info_ section in the OpenAPI document:
```
info:
  title: "<my service name>"
  description: |
    # Overview
    the quick brown fox jumps over the lazy dog
    ## Subsection
    the quick brown fox jumps over the lazy dog
  version: 0.0.7
```
In general, documentation entries **must not** contain references or links to internal documentation or internal concepts (e.g., internal headers like the [Dt-Tenant Header](../rest-api-guidelines/API%20Context%20Information.md#tenant-context)) even if they are relevant for the internal service implementation.

# OpenAPI Tag Usage
OpenAPI tags **should** be used to group logically, not technically. E.g., a service **should not** group all read operations to separate them from write operations. 

Tag descriptions **should** be used unless the tag name is self-explanatory. Tag descriptions **should** be short, and concise. You **should not** tag extensively; this might affect readability.

# Paths
- IAM Permissions **must** be shown for each path (this is generated by the API Gateway)
- Summary and/or description **must** be provided
    - For path endpoints
    - For parameters (query, path, header)
    - For request bodies
    - For response bodies
- Error responses **must** follow a [common format](../rest-api-guidelines/Common%20Schemas.md#error-response-format)
    - The [OpenAPI Template File](../rest-api-guidelines/template/openapi-template.yaml) already contains the most common error responses

# HTTP Response Codes
The API specification **must** always contain a default that covers all error codes. This can be either the [OpenAPI default entry](https://spec.openapis.org/oas/v3.0.3#fixed-fields-13) or these 2 OpenAPI range definitions: 4XX, 5XX. The default error responses **should** be represented by a (typically minimal) error envelope (see [Conventions](../rest-api-guidelines/Conventions.md#error-codes)).

Any response codes that have a service-specific meaning and all response codes representing a successful execution **must** be specified in addition to the default range definitions even if the response envelope is the same as the default.

There are several error codes which are considered common behavior of all services and therefore do not need to be documented individually on each service API over and over again. This reduces the size of the Swagger UI and gets rid of redundant documentation.

The following error codes are considered common knowledge (aka "no-brainer" codes) and **should not** be included individually in the documentation unless their usage deviates from the common way and/or contains service-specific information.

| Response Code	                  | Common Meaning                                                        |
| ------------------------------- | --------------------------------------------------------------------- |
| HTTP 400 - Bad Request	        | Should be omitted unless it contains specific information about the syntax error |
| HTTP 401 - Unauthorized	        | Generated by the IAM library, will be documented centrally            |
| HTTP 403 - Forbidden	          | Generated by the IAM library, will be documented centrally            |
| HTTP 404 - Not Found	          | **Should** be omitted if there is no explicit usage of 404 and 410 in the service or the response contains additional information about the failure.                                |
| HTTP 500 - Internal Error	      | Any unspecified internal error (often generated by some generic error handling in the service) |
| HTTP 501 - Not Implemented      | Method not supported, typically generated by the REST framework/the webserver. |
| HTTP 503 - Service Unavailable	| E.g., used when [throttling](../rest-api-guidelines/Design%20Patterns.md#rate-limiting-and-throttling) requests, **should** be documented if it contains specific retry information. **May** be omitted otherwise. |
 
Example see [OpenAPI template file](../rest-api-guidelines/template/openapi-template.yaml).

This allows the SDK code generators to build standard response handling code without the need to individually document each code over and over.
