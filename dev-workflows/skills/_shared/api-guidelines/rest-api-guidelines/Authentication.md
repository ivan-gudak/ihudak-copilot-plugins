# Authentication Method
Platform service APIs are not exposed to the public directly but via an API Gateway. The API Gateway handles the public authentication method which is [OAuth2.0 Client Credentials Flow](https://swagger.io/docs/specification/authentication/oauth2/) and provides a JWT token for platform services to use for authorization purposes within the platform. The API Gateway takes care of token refresh, token exchange and eventual mapping of browser session ids to internal JWT tokens. So, within the platform K8s cluster the only allowed method of authentication is the bearer JWT token generated by the API Gateway.

![Authentication](../rest-api-guidelines/img/authentication.png)

This means that while technically implementing a JWT bearer token as its [authentication context](../rest-api-guidelines/API%20Context%20Information.md#authentication-context), a platform service **must** provide the OAuth2.0 client credentials flow as the public [security schema](../rest-api-guidelines/OpenAPI.md) in its OpenAPI document. This is the authentication method that is seen and used by API clients while the JWT bearer token is the internal authentication mechanism only which must not be shown to the public. This also means that the `Authorization` header **must not** be defined in the OpenAPI document.

Other authentication methods **must not** be used or exposed in the OpenAPI document.

# IAM Permissions
All API endpoints **must** be protected by at least one IAM scope. In some rare cases an endpoint **may** use more than one scope, but this is typically avoided since it is confusing for customers.

The security schema defined in the [OpenAPI spec template](../rest-api-guidelines/template/openapi-template.yaml) is named `ssoAuth`, using it on an API endpoint looks like this:

```
/documents/{id}:
   get:
...
     security:
     - ssoAuth:
       - "document:documents:read"
```
Details on the IAM scope naming schema can be found in the [IAM Permission Guidelines](https://dynatrace.sharepoint.com/sites/Platform/SitePages/IAM-Permission-Guidelines.aspx).
